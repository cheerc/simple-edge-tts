## 1. Overview

**simple-edge-tts** is a cross-platform desktop application that converts text to speech using Microsoft Edge's TTS API. It provides a clean, modern GUI for selecting voices, adjusting parameters, previewing audio, and exporting MP3 files.

### Goals

- Replace the broken [edge-TTS-record](https://github.com/LuckyHookin/edge-TTS-record) tool
- Support both **macOS** and **Windows**
- Prioritize **English** and **Traditional Chinese (Taiwan)** voices
- Deliver as a single executable (`.exe` for Windows, `.dmg` for Mac)
- Published as a public repo at `github.com/cheerc/simple-edge-tts`

### Non-Goals

- Batch processing (multiple files at once) — future consideration
- Real-time streaming TTS — out of scope
- Voice cloning or custom model training — out of scope

## 2. Tech Stack

| Component | Choice | Rationale |
|---|---|---|
| Language | Python 3.11+ | edge-tts is Python; rich ecosystem |
| GUI Framework | PySide6 (Qt 6) | LGPL license (safe for public repo), native look, cross-platform |
| TTS Engine | edge-tts (pip) | Actively maintained, follows MS protocol changes |
| Audio Playback | QMediaPlayer (PySide6) | Built-in, no extra dependency |
| Packaging | PyInstaller | Single-file exe/app, proven cross-platform |
| Testing | pytest + pytest-qt | TDD, GUI testing support |
| CI/CD | GitHub Actions | Auto-build both platforms on release |

### Dependency: edge-tts

- Install as pip dependency (`pip install edge-tts`), not forked
- If Microsoft changes the API and edge-tts lags behind, fork at that point
- The `tts_engine.py` wrapper isolates all edge-tts calls — swapping implementations requires changes in one file only

### 3.2 Voice Selection

The voice dropdown groups voices into three priority tiers:

1. **🇹🇼 台灣中文** — `zh-TW-HsiaoChenNeural`, `zh-TW-YunJheNeural`, `zh-TW-HsiaoYuNeural`
2. **🇺🇸 English** — `en-US-*` voices (sorted by name)
3. **🌐 其他** — all remaining voices (collapsed/scrollable)

Features:
- Search/filter box at top of dropdown
- Display format: `Friendly Name (Language)` — e.g. `HsiaoChen (繁體中文)` / `HsiaoChen (Traditional Chinese)` depending on current language
- Remembers last used voice across sessions

### 3.3 Parameters

| Parameter | Control | Range | Default | edge-tts format |
|---|---|---|---|---|
| Rate (語速) | Slider + numeric label | -50% to +100% | 0% | `"+0%"` |
| Pitch (音調) | Slider + numeric label | -50Hz to +50Hz | 0Hz | `"+0Hz"` |

- Sliders show current value as label (e.g. `+20%`)
- Double-click slider to reset to default

### 3.4 Action Buttons

| Button | Behavior |
|---|---|
| ▶ **試聽 / Preview** | Generate TTS → play through speakers. Non-blocking (async). Button changes to ⏹ Stop during playback. |
| 💾 **匯出 MP3 / Export MP3** | Generate TTS → save to output folder. Shows progress. Filename: `{first_8_chars}_{YYYYMMDD_HHmmss}.mp3` |

- Both buttons are disabled when text input is empty
- During generation, show a progress indicator (indeterminate, since edge-tts doesn't report progress)

### 3.5 Theming

- **Follow system setting**: light theme when OS is in light mode, dark when dark
- Implemented via QSS (Qt Style Sheets) with two palettes
- No manual toggle — purely automatic via `QApplication.styleHints().colorScheme()`

### 3.6 Internationalization (i18n)

- Two languages: **zh-TW** (default) and **en-US**
- Toggle button in top-right corner: `繁中 / EN`
- All UI strings loaded from translation files (`zh-TW.json`, `en-US.json`)
- Language preference persisted in config
- Voice display names adapt to current language

## 4. Architecture

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  UI Layer   │────▶│  Controller  │────▶│  TTS Engine  │
│  (PySide6)  │◀────│  (signals)   │◀────│  (edge-tts)  │
└─────────────┘     └──────────────┘     └─────────────┘
       │                    │                     │
       │                    ▼                     │
       │            ┌──────────────┐              │
       │            │ Config Mgr   │              │
       │            │ (JSON file)  │              │
       │            └──────────────┘              │
       │                                          │
       ▼                                          ▼
┌─────────────┐                          ┌─────────────┐
│ Audio Player│                          │  Temp Files  │
│ (QMediaPlayer)                         │  (~/.cache)  │
└─────────────┘                          └─────────────┘
```

### 4.1 Module Responsibilities

| Module | Responsibility |
|---|---|
| `main.py` | Entry point, create QApplication, show window |
| `app.py` | Application-level setup: theme detection, high-DPI, single instance |
| `tts_engine.py` | Wraps edge-tts: list voices, generate audio (async). Returns file path. Isolates all edge-tts API calls. |
| `audio_player.py` | Play/stop audio files. State machine: idle → playing → idle. Uses QMediaPlayer. |
| `config_manager.py` | Read/write `~/.simple-edge-tts/config.json`. Provides defaults. Thread-safe. |
| `i18n.py` | Load translation JSON, provide `t("key")` lookup, emit signal on language change. |
| `ui/main_window.py` | Main window: layout, menu bar, status bar. Composes voice_panel + text_panel. |
| `ui/voice_panel.py` | Left panel: voice dropdown (with search), rate/pitch sliders, output folder picker. |
| `ui/text_panel.py` | Right panel: text input, action buttons, status display. |
| `ui/theme.py` | QSS definitions for light/dark themes. Listens to OS theme changes. |
| `update_checker.py` | Check GitHub Releases API for new version on startup. Non-blocking. |

### 4.2 Async Strategy

edge-tts is asyncio-based. PySide6's event loop is Qt-based. Solution:

1. Run edge-tts operations in a **QThread** with its own `asyncio.run()`
2. Emit Qt signals for progress/completion/error back to the UI thread
3. The `tts_engine.py` exposes synchronous-looking API that internally manages the async bridge

```python
class TTSWorker(QThread):
    finished = Signal(str)   # file path
    error = Signal(str)      # error message

    def run(self):
        asyncio.run(self._generate())
```

### 4.3 Config File

Location: `~/.simple-edge-tts/config.json`

```json
{
  "language": "zh-TW",
  "last_voice": "zh-TW-HsiaoChenNeural",
  "rate": "+0%",
  "pitch": "+0Hz",
  "output_dir": "~/Desktop",

```

- Created with defaults on first launch
- Written on app close and on each export
- Missing keys fall back to defaults (forward-compatible)

## 5. Filename Convention

Exported files follow: `{sanitized_text}_{timestamp}.mp3`

- `sanitized_text`: first 8 characters of input, non-filename-safe characters replaced with `_`
- `timestamp`: `YYYYMMDD_HHmmss` format
- Example: `你好世界測試用_20260624_173012.mp3`
- Example: `Hello_Wo_20260624_173012.mp3`

## 6.1 Auto-Update Notification

On startup, the app checks for new versions via the GitHub Releases API.

### Flow

1. **Background check** (non-blocking): `GET https://api.github.com/repos/cheerc/simple-edge-tts/releases/latest`
2. Compare `tag_name` (e.g. `v0.2.0`) with current version in app
3. If newer version available → show **non-modal notification** in status bar or dialog
4. If same or older, or network error → do nothing (silent)

### Update Dialog

```
┌────────────────────────────────────────────┐
│  🔄 有新版本可用                             │
│                                            │
│  目前版本：v0.1.0                            │
│  最新版本：v0.2.0                            │
│                                            │
│  ┌──────────────┐  ┌──────────┐            │
│  │ 前往下載頁面  │  │ 稍後提醒  │            │
│  └──────────────┘  └──────────┘            │
│                                            │
│  □ 不再提醒此版本                             │
└────────────────────────────────────────────┘
```

### Implementation

| Detail | Spec |
|---|---|
| Module | `update_checker.py` |
| Trigger | App startup, after main window shown |
| Threading | QThread (non-blocking, no UI freeze) |
| API | GitHub REST API (unauthenticated, 60 req/hr rate limit) |
| Comparison | Semantic versioning (major.minor.patch) |
| "Don't remind" | Store skipped version in config: `"skip_version": "0.2.0"` |
| Frequency | Once per launch (not periodic) |
| Button action | Open `https://github.com/cheerc/simple-edge-tts/releases/latest` in default browser |
| Error handling | Network error → silently ignore, log to console |

### Config Addition

```json

  "skip_version": null
}
```

## 7. Packaging & Distribution

### Windows
- PyInstaller `--onefile --windowed --icon=icon.ico`
- Output: `simple-edge-tts.exe`
- No installer needed, single exe
- ⚠️ **Known limitation**: Unsigned `.exe` triggers Windows SmartScreen warning. Users click "More info" → "Run anyway". Code signing requires an EV certificate (~$200-400/yr) — out of scope for v1.

### macOS
- PyInstaller `--windowed --icon=icon.icns`
- Package into `.app` bundle, then wrap in `.dmg` using `hdiutil`
- Output: `simple-edge-tts.dmg`
- ⚠️ **Known limitation**: Unsigned/unnotarized `.app` is blocked by Gatekeeper. Users right-click → "Open" → confirm, or run `xattr -cr simple-edge-tts.app`. Apple code signing + notarization requires Apple Developer Program ($99/yr) — out of scope for v1.

### README Workaround Instructions

The README must include clear instructions for bypassing security warnings:

**macOS**:
> 首次開啟時，macOS 會顯示「無法打開，因為無法驗證開發者」。
> 請在 Finder 中找到 app → 右鍵 →「開啟」→ 確認。
> 或在終端機執行：`xattr -cr /Applications/simple-edge-tts.app`

**Windows**:
> 首次執行時，Windows SmartScreen 會顯示「Windows 已保護您的電腦」。
> 請點擊「其他資訊」→「仍要執行」。

### 8.3 `.github/workflows/ci.yml` — PR Gate

- **Trigger**: PR to `main`, paths-ignore `*.md` and `docs/**`
- **Permissions**: `contents: read` (read-only, 最小權限)
- **Concurrency**: `group: pre-merge-checks-${{ github.event.pull_request.number || github.ref }}`, cancel-in-progress
- **Steps**: checkout → setup-python → pip install → `./workflow.sh t2` → `t3` → `t4` → `t5` (individual steps for CI visibility)
- **Timeout**: 10 minutes
- **Security**: Actions pin 到 commit SHA（非浮動 tag，避免 supply chain attack）
- **Note**: CI runs each step separately (not `t6`) so GitHub Actions shows which exact step failed

### 8.4 `.github/workflows/build.yml` — Release Build

- **Trigger**: push tag `v*`

- **Jobs**: `build-windows` (windows-latest) + `build-macos` (macos-latest)
- **Each job**: checkout → setup-python → pip install → pytest → PyInstaller → upload artifact
- **Release job**: download artifacts → auto-generate CHANGELOG from `git log` since last tag → create GitHub Release with `.exe` + `.dmg` + changelog body
- **Security**: Actions pin 到 commit SHA（同 ci.yml）

## 9. Testing Strategy (TDD)

Tests are written **before** implementation. Order follows dependency graph:

| Phase | Module | Test Focus |
|---|---|---|
| 1 | `config_manager` | Read/write config, defaults, missing keys, corrupt file recovery |
| 2 | `i18n` | Translation loading, key completeness (both languages have same keys), fallback |
| 3 | `tts_engine` | Voice listing, parameter formatting, async generation (mock edge-tts) |
| 4 | `audio_player` | State transitions: idle→playing→idle, stop during play, file not found |
| 5 | `ui/*` | Widget rendering, button states, slider values, language switch (pytest-qt) |

### Test Dependencies
- `pytest` — test runner
- `pytest-qt` — PySide6 widget testing
- `pytest-asyncio` — async test support
- `unittest.mock` — mock edge-tts network calls

### Test Conventions
- All tests in `tests/` mirror `src/` structure
- Network calls are always mocked (tests run offline)
- Each test file has a docstring explaining what module it covers

## 10. Development Workflow

Solo-adapted development lifecycle for a public repo.

### 10.1 Issue-First

- Every non-trivial change starts with a GitHub issue
- PR body uses `Closes #N` or `Refs #N` to link issues
- Trivial fixes (typos, docs) can skip issue creation

### 10.2 Branch Model

**main + feature branches**:
- `main` is the release branch, always deployable
- Development on `feat/xxx`, `fix/xxx`, or `chore/xxx` branches
- Branch protection on `main`: require CI pass, no direct push
- Flow: create branch → develop (TDD) → `./workflow.sh t6` local green → push → PR → CI green → merge

### 10.3 Verification Before Completion

Before claiming any work is done or any test is passing:
- Actually run the command and confirm the output
- Paste the real output as evidence in commit messages or PR body
- Never say "tests pass" without running `./workflow.sh t6` first

### 10.4 Merge Gate

- CI must pass (enforced by branch protection)
- Self-review the PR diff before merging
- Squash-merge to keep main history clean

### 10.5 Post-Merge Cleanup

- Delete the feature branch after merge
- Verify `main` is green after merge

### 10.6 Release Flow

1. `./deploy.sh v1` — bumps version in `pyproject.toml`, commits, creates git tag
2. `git push origin main --tags` — triggers `build.yml`
3. CI builds Windows `.exe` + macOS `.dmg`, auto-generates CHANGELOG, creates GitHub Release
4. Verify the Release page has both binaries attached

### 10.7 Retrospective

After each milestone (e.g. v0.1.0, v0.2.0):
- Note lessons learned in the release body or a `CHANGELOG.md` entry
- Lightweight: 2-3 bullet points on what went well / what to improve
- No formal retro process needed for solo development
